## 1. 项目目标

从历年高考压轴题中提炼、总结各种难题的解法,沉淀为一个**结构化的解题方法库**,并在其上提供两类消费应用。

分层定位:

- **核心(C)· 解题方法库**:方法 → 适用特征 → 一组例题。可反复查询的知识图谱,是系统的地基与最终产物。
- **应用 A · 教师/教研报告**:生成"解法专题总结"报告(如"导数压轴题的主流解法+适用条件"),供人阅读与二次编辑。
- **应用 B · 学生问答助手**:学生贴题 → 给出解法思路、点明所用套路、关联同类例题。

A、B 均建立在 C 之上,只读 C 中**已审核确认**的数据。

## 2. 范围与约束

- **输入**:第一版只消费干净文本/LaTeX。架构预留图像入口;未来 OCR 直接接**腾讯云/阿里云数学公式识别 API**,不自研(YAGNI)。
- **交付形态**:核心引擎优先(架构上)+ 第一版即带 Web 界面(交付上)。核心逻辑封装为独立、可复用、可独立测试的 Python 模块;Web 为薄壳。
- **LLM/嵌入**:DeepSeek 默认 + 可配置 LLM 接口层。核心逻辑面向抽象接口编程,换模型只改配置。
- **存储**:SQLite(结构化)+ Chroma(向量)起步,零运维;数据访问封装在 Repository 接口后,将来可换 PostgreSQL + pgvector 而不动业务逻辑。
- **前置输入(需教研提供)**:一份初始的高考数学解题方法清单作为 taxonomy 种子。第一版可先覆盖 1–2 个板块(如导数、圆锥曲线),系统随审核逐步扩充。

## 3. 整体架构

三层,自下而上。核心引擎层**完全不依赖 Web**;Web 与未来的 CLI 都只是调用者。

```
┌─────────────────────────────────────────────────────┐
│  交付层 (Web 薄壳)  FastAPI + 前端                    │
│  · 题目录入/审核界面  · 方法库浏览  · 报告  · 问答     │
├─────────────────────────────────────────────────────┤
│  核心引擎层 (纯 Python,可独立测试,不依赖 Web)        │
│   提炼 Extractor · 聚类 Clusterer · 报告 Reporter ·   │
│   问答 QA                                             │
│              ┌────────────────────────┐               │
│              │ LLM 接口层 (默认 DeepSeek)             │
│              │ Embedding 接口层        │               │
│              └────────────────────────┘               │
├─────────────────────────────────────────────────────┤
│  存储层 (Repository 抽象)                              │
│  ProblemRepo · MethodRepo · SolutionRepo · VectorRepo │
│  · SQLite (结构化)   · Chroma (向量)                  │
└─────────────────────────────────────────────────────┘
```

**核心原则:**

- 核心引擎接受普通 Python 对象/参数,返回结果对象,不知道 Web 的存在。
- LLM 与 Embedding 藏在接口层后;业务代码调抽象方法,不关心具体后端。
- 所有数据访问走 Repository;业务代码不关心底层是 SQLite 还是 Chroma。
- 第一版最该验证的"提炼+聚类质量",可脱离 Web、由测试直接驱动核心引擎跑通。

## 4. 数据模型

方法库是一张知识图谱,三个核心实体。

### 4.1 Problem(题目)

| 字段 | 说明 |
|------|------|
| id | 唯一标识 |
| year, region | 年份、地区(如 2023 全国甲卷) |
| subject_area | 知识板块(导数/圆锥曲线/数列/不等式…) |
| stem_latex | 题干(LaTeX/文本) |
| reference_solution | 参考答案原文(可选) |
| source | 来源标注 |
| content_fingerprint | 内容指纹,用于去重/幂等 |
| image_ref | 图像入口预留字段(第一版留空) |
| created_at | |

### 4.2 Method(解题方法)

taxonomy 骨架中的方法节点,是"总结"的产物。

| 字段 | 说明 |
|------|------|
| id | |
| name | 方法名(如"分离参数法""切线放缩") |
| subject_area | 所属板块 |
| parent_id | 层级(大类→具体手法),可空 |
| applicability | 适用特征——什么样的题该用它(结构化描述) |
| core_idea | 核心思想/为什么有效 |
| procedure_steps | 通用步骤模板 |
| pitfalls | 常见坑/易错点 |
| status | `seed`(预置)/ `confirmed`(已审核)/ `candidate`(待确认新方法) |
| created_at, updated_at | |

### 4.3 SolutionInstance(解法实例)

连接题与方法的"边":记录"这道题是怎么用这个方法解的"。

| 字段 | 说明 |
|------|------|
| id | |
| problem_id | → Problem |
| method_ids | → Method(可多个:一题可用多方法组合) |
| key_steps | 这道题里该方法的关键步骤(具体演绎) |
| transfer_note | 可迁移的套路提炼(从这道题学到的通用招法) |
| embedding_id | → VectorRepo 中该解法描述的向量 |
| confidence | LLM 自报置信度 + 系统归类置信度 |
| review_status | `draft`(LLM草稿)/ `confirmed`(人工确认或自动确认)/ `rejected` |
| reviewer_note | 审核备注 |
| llm_raw | LLM 原始输出留存(可追溯) |
| created_at | |

**设计要点:**

- Method 与 SolutionInstance 分离,才能实现"方法 → 一组例题":一个 Method 挂多个 SolutionInstance;报告即"某方法 + 它的一组典型例题"。
- 审核状态落在 SolutionInstance 与 Method 上,承载人工审核工作流。
- embedding 挂在 SolutionInstance(解法)而非 Problem(题目)上——聚类的是解法;同一题的不同解法应落入不同方法簇。
- 保留"一题多方法":压轴题常是多种方法组合,这正是要总结的精华。

## 5. 核心管线:Ingest → Extract → Classify → Review → Commit

一道题从录入到入库的 5 步管线,每步职责单一、可独立测试。

### ① Ingest(录入)
- 消费干净文本/LaTeX,解析成 `Problem`,存库(状态:待提炼)。
- 计算 `content_fingerprint`,重复录入可识别,避免重复提炼(幂等)。
- 图像入口在此预留:未来先过腾讯云/阿里云公式识别 API,输出文本后汇入同一入口。

### ② Extract(LLM 提炼)
- 题干 + 参考答案 → LLM(DeepSeek),用**结构化输出(强制 JSON schema)** 产出:所用方法、每个方法的关键步骤、可迁移套路、适用特征、自报置信度。
- 产出 `SolutionInstance(review_status=draft)` + `llm_raw` 留存。
- **prompt 注入当前 taxonomy 方法清单**,引导 LLM 优先归入已有方法,而非每题自创命名——保证体系一致性的核心手段。
- schema 解析失败 → 有限次重试 → 仍失败降级为"可疑项"进审核队列,绝不静默丢弃。

### ③ Classify(归类 + 发现新方法)
- LLM 已归入已有 Method → 直接挂上。
- LLM 提出的方法名不在体系内 → 将解法描述向量化,在 VectorRepo 查最相似的已有解法:
  - 相似度**高于上阈值** → 建议归入该簇对应 Method。
  - 相似度**低于下阈值(离群)** → 标记为候选新方法 `Method(status=candidate)`,进审核队列。
  - 落在**中间模糊带** → 标记为可疑项,进审核队列。
- 机制价值:预置体系挡噪音,离群题浮出成为新方法——体系稳定又能进化。

### ④ Review(人工审核)——"只审可疑项"
- **第一版策略:只审可疑项,非全审。** 高置信 + 干净归入已有方法 → 自动 `confirmed`。
- **可疑项判定(可配置)**:
  1. LLM 提出的方法不在 taxonomy;
  2. 嵌入相似度落在中间模糊带;
  3. LLM 自报低置信,或结构化输出解析异常;
  4. 一题被判定用了过多方法(超阈值,疑似 LLM 发散)。
- 审核队列展示:LLM 草稿、归类建议、相似例题对比、离群提示。
- 审核员动作:确认 / 改写方法归属 / 修订步骤与套路 / 批准候选新方法 / 拒绝。动作落到 `review_status`、`Method.status`。
- **未确认内容不进入报告与问答**,消费端只见 confirmed 数据。

### ⑤ Commit(固化)
- 确认后写入向量库、更新 Method 与例题关联、刷新方法聚合统计(如"该方法现有 N 个例题")。

**可配置严格度**:相似度上下阈值、方法数上限、是否收紧/放宽可疑判定,均为配置项。第一版按"只审可疑项"运行;质量稳定后可进一步放宽。

## 6. 消费应用

两者均**只读**核心库,且**只读 confirmed 数据**。

### 6.1 应用 A · 教师报告(Reporter)
- 输入:一个方法或一个板块(如"导数压轴题"或"分离参数法")。
- 过程:取出该方法(及子方法)+ 其下全部 confirmed 例题 → LLM 组织成结构化专题报告。
- 输出:解法专题总结——核心思想、适用特征、通用步骤、典型例题(按难度/年份/变式排列)、常见坑。可导出 Markdown。
- 本质:方法库已结构化知识,Reporter 只做"取数 + LLM 润色成文"。

### 6.2 应用 B · 学生问答(QA,基于 RAG)
- 输入:学生贴一道题,或提问"这类题怎么做"。
- 过程:
  1. 对题目做与 Extract 相同的特征提取 + 向量化;
  2. 在库中检索最相似的解法簇 / 方法(RAG 检索);
  3. LLM 结合"检索到的方法知识 + 这道具体题"生成回答:该用哪种套路、为什么(适用特征匹配)、关联同类例题。
- 关键:回答**基于已沉淀的方法库(RAG)**,而非纯 LLM 裸答——体现方法库价值,更可控、可溯源。
- **边界**:QA 只读、不自动写库。学生贴的题若有价值,走"建议入库"旁路,仍须经正常提炼+审核管线;消费端绝不污染核心库。

## 7. 错误处理

- **LLM 输出不可信任**:结构化输出强制 schema 校验;解析失败→有限次重试→仍失败降级为可疑项进审核,绝不静默丢弃或写脏数据。
- **外部 API 边界**:LLM/嵌入调用统一封装,含超时、重试、限流退避;失败有明确错误信息,不吞异常。
- **输入校验**:录入在边界校验(必填字段、LaTeX 基本合法性),脏数据挡在入口外。
- **幂等**:按来源 + 内容指纹识别重复录入,避免重复提炼。

## 8. 测试策略(TDD,目标覆盖率 80%+)

- **单元测试**:每个引擎模块(Extract/Classify/Reporter/QA)独立测,**LLM 与 Embedding 用 mock**——不依赖真实 API,快速、可确定性断言。
- **集成测试**:Repository 层对真实 SQLite/Chroma 跑;管线端到端(mock LLM)验证状态流转(draft→confirmed、candidate 升级等)。
- **契约测试**:LLM 接口层用少量真实调用验证"prompt 能拿到合规结构化输出",单独标记、不进常规 CI。
- 采用 AAA(Arrange-Act-Assert)结构;测试名描述行为。
